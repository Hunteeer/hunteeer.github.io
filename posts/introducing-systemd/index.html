systemd 在 CentOS 中是很庞大的一部分，很多 Linux 发行版都整合了 systemd，而且和 macOS 上的 launchd 有点类似。

Introduction

systemd 是 Linux 下的一款系统和服务管理器，init system。到 2015 年，大部分发行版已经将 systemd 作为其默认的 init system，取代了 SysV 和 LSB initscripts。

开发者

systemd 由 Red Hat 公司软件工程师 Lennart Poettering 和 Kay Sievers 开发，最初目的是想提高系统启动的效率。systemd 很多概念来源于 macOS 上的 launchd（确实挺像，至少在使用上）[1]。Poettering 后来进一步定义 systemd 的设计目标是通过提供以下三个基本功能”实现发行版之间的无差别”[2]：

  • 一个系统和服务的管理器：通过应用各种配置管理系统和服务
  • 一个软件平台：作为开发其他软件的基础服务
  • 应用进程和内核之间的胶水：提供各种内核提供的功能的接口

重要特性[1][3]

  • 兼容性：兼容 SysV 和 LSB 启动脚本，使 systemd 能够取代之前的 init system 工作，而不对现有软件造成过多影响
  • 并行启动：systemd 将大量的启动工作解除了依赖，使得它们可以并发启动。加快系统启动速度
  • 按需启动：某个服务被真正请求时才会启动它，服务结束时停止它
  • 采用 Linux 的 Cgroup 特性跟踪、管理进程的生命周期
  • 维护挂载点和自动挂载

基本工具

systemd 是 PID=1 的系统守护进程（deamon），最先被启动。值得注意的是，systemd 不仅仅指的是这个守护进程，它还包括一系列 daemon ，比如 journald、logind、networkd，和低级别组件。Poettering 将其定义为”软件套装“（全，，全家桶？）。

systemctl

这是查看和监控 systemd，管理系统和服务的命令。下面是Systemd 命令和 SysVinit 命令的对照表[1]。

Sysvinit 命令 Systemd 命令 备注
service foo start systemctl start foo.service 用来启动一个服务 (并不会重启现有的)
service foo stop systemctl stop foo.service 用来停止一个服务 (并不会重启现有的)。
service foo restart systemctl restart foo.service 用来停止并启动一个服务。
service foo reload systemctl reload foo.service 当支持时，重新装载配置文件而不中断等待操作。
service foo condrestart systemctl condrestart foo.service 如果服务正在运行那么重启它。
service foo status systemctl status foo.service 汇报服务是否正在运行。
ls /etc/rc.d/init.d/ systemctl list-unit-files –type=service 用来列出可以启动或停止的服务列表。
chkconfig foo on systemctl enable foo.service 在下次启动时或满足其他触发条件时设置服务为启用
chkconfig foo off systemctl disable foo.service 在下次启动时或满足其他触发条件时设置服务为禁用
chkconfig foo systemctl is-enabled foo.service 用来检查一个服务在当前环境下被配置为启用还是禁用。
chkconfig –list systemctl list-unit-files –type=service 输出在各个运行级别下服务的启用和禁用情况
chkconfig foo –list ls /etc/systemd/system/*.wants/foo.service 用来列出该服务在哪些运行级别下启用和禁用。
chkconfig foo –add systemctl daemon-reload 当您创建新服务文件或者变更设置时使用。
telinit 3 systemctl isolate multi-user.target (OR systemctl isolate runlevel3.target OR telinit 3) 改变至多用户运行级别。

此外，systemd 也包含电源管理的功能，支持的命令[1]：

命令 操作
systemctl reboot 重启机器
systemctl poweroff 关机
systemctl suspend 待机
systemctl hibernate 休眠
systemctl hybrid-sleep 混合休眠模式（同时休眠到硬盘并待机）

journalctl

这是管理 systemd 日志系统的命令。systemd 是用二进制格式保存日志的，用户要使用 journalctl 命令来读取日志信息。journald 的配置文件是 /etc/systemd/journald.conf，日志文件保存在 /var/log/journal，如果该目录不存在，systemd 会将日志保存在 /run/systemd/journal，意味着系统重启后日志将会消失[3]。

查看一下是否如此：

$ ls -l /run/systemd/journal
total 4
-rw-r--r--. 1 root root 0 Apr 28 14:33 flushed
-rw-r--r--. 1 root root 8 Apr 28 14:33 kernel-seqnum
srw-rw-rw-. 1 root root 0 Apr 28 14:33 socket
srw-rw-rw-. 1 root root 0 Apr 28 14:33 stdout

常用的命令有：

$ journalctl -u netcfg	#显示指定单元的日志
$ journalctl _PID=1	#显示指定进程的日志
$ journalctl -f	#显示当前日志(动态)
$ journalctl -b	#显示本次启动后的所有日志
$ journalctl -b -1	#显示上次启动的所有日志
$ journalctl -p err	#显示特定级别的日志(err 级)
$ journalctl -p 3	#显示特定级别的日志(err 级)，和上一条等同

单元文件的编写

systemd 的功能繁多，不仅用来管理服务，还可以管理挂载点，定义定时任务等。这些工作都是由编辑相应的配置单元文件完成的[1]。单元文件（unit）就是配置文件，用来保存各种类型的单元的信息，方便 systemd 作统一的管理。这样做对使用者来说其实更加方便。单元文件存放的目录主要有（优先级从高到低）[4]：

Path Description
/etc/systemd/system Local configuration
/run/systemd/system Runtime units
/usr/lib/systemd/system Units of installed packages

单元文件主要有以下几个种类：

  • service：encodes information about a process controlled and supervised by systemd
  • socket：encodes information about an IPC or network socket or a file system FIFO controlled and supervised by systemd, for socket-based activation
  • device：encodes information about a device unit as exposed in the sysfs/udev(7) device tree
  • mount：encodes information about a file system mount point controlled and supervised by systemd
  • automount：encodes information about a file system automount point controlled and supervised by systemd
  • swap: encodes information about a swap device or file for memory paging controlled and supervised by systemd
  • target: encodes information about a target unit of systemd, which is used for grouping units and as well-known synchronization points during start-up
  • path: encodes information about a path monitored by systemd, for path-based activation
  • timer: encodes information about a timer controlled and supervised by systemd, for timer-based activation
  • slice: encodes information about a slice which is a concept for hierarchically managing resources of a group of processes

单元文件的编写分为三个部分，以下两个部分是所有的单元文件都有的：

  • [Unit]：单元文件的声明
  • [Install]：安装服务

此外，还有一个部分可以用来明确单元文件的类别，是最主要的。比如 service 单元就有一个 [Service]。

编写一个 .service 单元文件[5]

.service 单元文件的编写是最常用到的。单元文件的语法格式属于声明性语言。以上一篇博客 aria2，目前我认为最好的下载工具 提到的 aria2 的单元文件为例。

[Unit]

是单元文件的声明，包含了对单元文件的描述：

[Unit]
Description=The next generation download utility	#描述可以随便写

[Unit] 还包含一些单元之间的依赖关系：

Requires=	#对添加的单元具有依赖性。只有需要的单元运行，它才能运行
Wants=	#推荐使用。本单元启动了，它“想要”的单元也会被启动。但是启动不成功，对本单元没有影响
Conflicts=	#一个单元的启动会停止与它“冲突”的单元

还能规定单元之间的启动顺序：

Before=	#在"Before"的单元之前启动。
After=network.target	#在"After"的单元之后启动。在例子中，aria2 是下载工具，很明显需要网络的有效性，所以安排在 network.target 之后启动。但是就算还是没网，aria2.service 也还是会启动

注意是先启动而不是启动成功，因为失败也是一种状态，一定要成功才启动另一个服务是通过依赖关系定义的。反之 After 亦然

[Service]

定义服务的本体。

Type=	#声明服务的类型。有 simple, forking, oneshot, dbus,notify, idle。默认是 simple
PIDFile=	#指明 PID 文件的绝对路径。推荐当 service 的 Type=forking 时使用
ExecStart=/usr/bin/aria2c --conf-path=$HOME/.aria2/aria2.conf	#启动服务的命令。除了 oneshot 之外，必须且只能给出一行命令来启动服务。在例子中给出了以配置文件启动 aria2 的命令
ExecStartPre=	#启动服务之前预先需要执行的命令

更多详细信息参考 man systemd.service(5)

[Install]

在指定位置放置并编写了单元文件，并不代表就能够用了。比如在例子中 aria2.service 放置在 /usr/lib/systemd/system。还需要在 systemd 内部对它进行加载。

[Install]
WantedBy=multi-user.target	#表示本服务被多用户系统所需要。这样在 multi-user.target 启用时，服务也就会被启用
Alias=	#设置别名。使用 systemctl 的时候可以使用别名
Also=	#安装本服务还需要安装什么别的服务。它管理的不是运行时候的依赖，而是安装的时候
